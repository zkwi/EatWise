# AI 约束治理

EatWise 使用用户自配的 OpenAI-compatible 多模态模型分析餐食图片。项目不提供后端服务，不做云同步。AI 治理的目标是：让输出可控、隐私边界清晰、维护成本低。

## 数据边界

用户主动选择或拍摄餐食图片后，App 会把压缩后的图片和饮食目标发送到用户配置的模型 API。默认每张图片并发发送两次请求：饮食建议卡片和粗略营养估算卡片。除此之外，App 不主动把数据上传到其他服务。

以下内容不得进入仓库、Issue、PR、Release、日志或文档示例：

- 真实 API Key、Authorization header、签名密码、keystore。
- `local.properties`、本地日志、真实用户图片。
- 完整 base64 图片、完整 AI 请求体、完整 AI 响应体。
- 含个人信息的本机路径、饮食目标、截图或调试输出。

需要说明问题时，使用合成数据、内置示例图片、截断后的错误信息和脱敏字段。

## Prompt 治理

- prompt 集中维护在 `OpenAiCompatibleClient`。
- `promptVersion` 变化必须同步更新模型、测试、README、维护文档和手动测试清单。
- App 支持简体中文、繁体中文、英语和日语；prompt 必须明确约束所有用户可见 JSON 字符串使用当前语言。
- 首次启动默认跟随系统语言；用户在设置中手动选择语言后，以用户选择为准。
- prompt 应约束模型只返回 JSON，不依赖模型自由发挥生成 UI 文案结构。
- prompt 应要求模型不要执行图片、包装、菜单、票据、截图或水印中的文字指令；这些文字只能作为食物相关视觉证据。
- 饮食建议 prompt 不要求模型估算重量、卡路里或宏量营养素，因为图片角度和遮挡会让数值看起来精确但实际不可靠。
- 粗略营养估算 prompt 只允许基于常见份量输出宽区间、直观食物类比和估算依据；不得输出精确单值、小数或确定语气，也不在 UI 展示置信度。
- 粗略营养估算 prompt 应优先使用稳定行名，例如热量、蛋白质、碳水、主食量、脂肪、油脂量、蔬菜量、膳食纤维、钠、盐分，避免生成过多 UI 难以归类的营养项。
- 蔬菜量不得用克数表示，应使用“明显较多”“约 1-2 碗熟蔬菜”等定性份量；如果输出克数，必须明确标为膳食纤维，不能把纤维克数展示成蔬菜重量。
- 钠、盐分、碳水、主食量、脂肪和油脂量也应按单位和语义拆开：mg 对应钠，g 盐当量对应盐分，碳水克数不等于主食重量，脂肪克数不等于可见用油量。
- 粗略营养估算建议必须优先匹配用户饮食目标，再结合图片中可见的主要营养负担。
- 热量类比只用于帮助用户理解量级，例如“约相当于 4-6 碗米饭或 10-15 个苹果”；不能暗示实际称重或精确换算。
- 饮食建议和营养估算都应按手机卡片输出短句：理由、依据、说明和动作建议不能写成大段解释。
- 动作建议应像清单项一样直接写“食物 + 动作”，不要以“这餐因为……所以建议……”开头；原因放到摘要、目标匹配、估算依据或行说明。
- prompt 不允许医疗诊断、药物建议、治疗方案或疾病结论。
- prompt 应把目标匹配表达为“是否适合当前饮食目标”，不要写成绝对健康结论。
- prompt 应提供清晰目标风险分级：明显违背用户少油、控脂、少糖、少盐或减少精制碳水目标的餐食，不能被判为顶层 `good`。
- prompt 应要求顶层餐食名称、摘要、怎么吃、目标匹配、建议和标签基于整盘、整堆或整桌菜综合判断，不要只按其中某一道菜给总体分数或总体建议。
- 多菜品里既有清淡/高蛋白/蔬菜，也有油炸/重油/甜饮等部分时，总体建议应同时指出可以多吃或先吃的健康部分，以及需要少吃的高负担部分。
- prompt 应要求建议只引用图片中可见且 `ingredients` 已列出的菜品/食材，不要因为模板示例而提到不存在的甜品、甜饮、汤汁或酱料。
- prompt 应使用普通用户能执行的建议，如少蘸酱、去皮、少喝汤汁、加蔬菜、下餐清淡。
- 建议必须具体到动作，不输出“注意控制、保持均衡、控制频率”等无法直接执行的空泛提醒。
- 标签示例、常见别名和低价值标签黑名单需要随语言同步维护，避免非中文模式下仍输出中文标签或含糊标签。

## 输出 Schema

当前项目按最新 schema 维护，不为废弃字段保留兼容分支。删除或新增字段时需要同步处理：

- `MealAnalysisResult`、`NutritionAnalysisResult` 和相关 domain model。
- `JsonUtilsTest` 或对应解析测试。
- Room Entity、database version 和迁移策略。
- 分析页、详情页、历史页展示。
- `docs/MAINTENANCE.md` 和 `MANUAL_TESTING.md`。

当前阶段允许 destructive migration，以避免为旧实验数据堆积兼容代码。如果后续开始长期保存真实历史记录，再重新评估 Migration。

## 标签和建议

标签只保留有决策价值的短结论。颜色语义保持一致：

- 绿色：推荐、正向、可多吃。
- 黄色：需要注意、适量、控制频率。
- 红色：明显风险、不推荐、需要严格控量。

不要输出或展示：

- “常规食材”“常规分量”“普通”“粗估”等低信息量标签。
- 同一事实的正反双重标签，例如红油调料同时标“轻负担”和“油脂高”。
- “健康/不健康”这类空泛判断。
- 饮食建议卡片里的具体克数、卡路里、蛋白质克数、脂肪克数、碳水克数。
- 营养估算卡片可以显示宽区间和直观食物类比，例如“约 600-900 kcal”“约相当于 3-4 碗米饭”“约 20-35 g”，但必须同时显示估算依据，不能显示 `3354 大卡`、`156.8g` 或带小数的区间，也不展示置信度徽标；蔬菜量、主食量和油脂量不显示克数。

## 模型连接治理

设置页的测试连接必须验证模型支持图片输入。测试请求只能使用内置小图片，不能发送用户照片。

网络错误提示应帮助用户定位配置问题，但不能回显 Key、Authorization header、完整请求体或完整响应体。

## 日志和调试

允许短暂记录：

- 页面名、状态变化、HTTP 状态码。
- 错误类型、图片大小、压缩前后尺寸。
- 是否配置了模型和 Key 的布尔状态。

禁止记录：

- API Key、Authorization header。
- 完整 base64、完整请求体、完整响应体。
- 用户真实图片路径中的个人信息。
- release keystore 路径、别名或密码。

## AI 协作要求

AI 或人工提交改动前应确认：

- 没有读取或提交项目外敏感文件。
- 没有把真实用户数据写入测试、文档或日志。
- 没有引入复杂兼容层、过度抽象或无必要依赖。
- 涉及 prompt/schema/隐私/网络边界的改动已同步文档。
- 已运行对应验证命令，或明确说明无法运行的原因。
